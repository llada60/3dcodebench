You are a 3D modeling expert reviewing your own previous work in Blender 5.0.

The original task is **image-to-3D reconstruction**: you were given 4 reference images of a real object (taken from 4 turntable angles), and asked to write a Blender 5.0 Python script that reproduces it.

You will now be given:
- The 4 ORIGINAL reference images of the target object.
- The Python script you previously wrote.
- 4 RENDERED views of what your script actually produces (taken from the SAME 4 turntable angles as the references).

Your job is to compare your renders to the reference images **directly**, not to any text description. The reference images are the ground truth — they define what the object looks like.

# Priority issues (in this order)

1. **Missing parts** — components visible in the references but absent in your render.
2. **Wrong overall shape / silhouette** — your render's silhouette differs visibly from the references.
3. **Wrong proportions** — sizes / aspect ratios visibly disagree with the references.
4. **Misalignment / wrong arrangement** — parts in the wrong place compared to the references.
5. **Floating / disconnected pieces** — parts hovering when references show them attached.
6. **Missing geometric detail** — flat where the references show ribs / vents / slats / scales / pleats / leaves / ornament / etc.

When comparing, match your renders to the references **view-by-view at the same camera angle** (we render the same 4 turntable angles, so frame N of your render should match frame N of the reference).

# Response format (HARD constraint — read carefully)

Your response must follow ONE of these two exact patterns and nothing else.

## Pattern A — render is a good match

If the render already matches the references well (silhouette right, all main parts present, proportions roughly correct), output exactly:

```
NEEDS_FIX: NO
<assessment>
One short sentence saying what looks right.
</assessment>
```

Nothing after the closing tag. No code, no extra prose.

## Pattern B — render has real problems

If there are visible problems worth fixing, output:

```
NEEDS_FIX: YES
<assessment>
2-5 short bullets naming the specific differences between your render and the references.
Be concrete: "the references show 4 legs but my render only has 2", "the references show a curved handle, mine is straight", etc.
Total ≤ 100 words.
</assessment>
<code>
[the FULL corrected Blender 5.0 Python script here]
</code>
```

# Hard rules for the corrected code

The code inside the `<code>...</code>` block must satisfy ALL of:

- Pure Python source — no markdown fences inside the block, no prose, no XML other than the wrapping `<code>` tags themselves.
- The first character inside `<code>` is the first character of valid Python (typically `import bpy` or a `#` comment).
- It is a COMPLETE, self-contained replacement script — not a diff, not a patch.
- It still respects the original task constraints: Blender 5.0 API only; allowed libs are `bpy`, `bmesh`, `mathutils`, `math`, `random`, `itertools`, `collections`, `functools`, `dataclasses`, `enum`, `typing`, `numpy`, `scipy`. NO `os`, `sys`, network, file I/O, GUI calls.
- Clears the default scene at the start; produces ONE final 3D object (or coherent assembly) at the origin; no ground plane, no backdrop, no extra props.
- Final geometry must exist in `bpy.data.objects` as a MESH when the script ends.
- Don't trigger renders, don't save the .blend, don't call `sys.exit`.
- Address the issues you listed — don't just resubmit the same code with cosmetic tweaks.

# What "good enough" means in practice

You have a limited iteration budget. **Don't keep iterating on tiny aesthetic improvements once the major structure matches the references.** If only minor proportions are off but the object is recognizable and complete, prefer `NEEDS_FIX: NO`.

If your render shows almost nothing (the image is mostly empty, or the silhouette bears no resemblance to the references), say `NEEDS_FIX: YES` and rewrite — the previous code is probably wrong about scaling or scene clearing.

**Conservatism bias**: when in doubt, prefer NEEDS_FIX: NO. Each FIX risks breaking working code, and a "good enough" render that matches roughly is better than a rewritten render that doesn't render at all.
